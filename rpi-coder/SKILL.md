# RPI Coder — Java Code Review & Refactoring

基於 Context Engineering 原則（來自 Dex Horthy 的 "No Vibes Allowed" 方法論），
讓 AI 在複雜的 Java codebase 中進行高品質的 code review 與重構。

核心信念：**context 的品質決定輸出的品質**。不急著動手，先理解，再計畫，最後才實作。

---

## 三階段流程

### Phase 1：Research（理解系統）

**目標：** 在不修改任何程式碼的前提下，建立對問題範圍的完整理解。

#### 判斷是否需要 Sub-agent

先評估探索規模，再決定執行方式：

| 情境 | 做法 |
|------|------|
| 使用者直接貼上程式碼 | 直接分析，不需要 sub-agent |
| 指定 1-2 個明確的檔案 | 直接讀取，不需要 sub-agent |
| 只提供類別名稱或功能描述，需要在 codebase 中尋找 | **啟動 Sub-agent 探索** |
| 涉及 3 個以上相互依賴的模組 | **啟動 Sub-agent 探索** |

#### 不需要 Sub-agent 時（直接執行）

1. **識別範圍** — 釐清使用者要 review 或重構的是哪些檔案/類別/功能
2. **閱讀相關程式碼** — 找出並閱讀：
   - 目標類別本身
   - 它依賴的介面與關鍵實作
   - 呼叫它的上層程式碼（caller）
   - 相關的測試檔案
3. **理解設計意圖** — 從命名、註解、測試推斷這段程式碼的職責
4. **記錄發現** — 整理成研究文件（見下方格式）

#### 需要 Sub-agent 時（大型 codebase 探索）

啟動一個 sub-agent，任務指令如下：

```
你是一個 Java codebase 探索 agent。
任務：找出與 [使用者描述的功能/類別] 相關的所有程式碼。

請執行：
1. 搜尋相關的類別、介面、方法名稱
2. 找出呼叫鏈（從 Controller/Service 到 Repository）
3. 找出對應的測試檔案

只回傳以下內容，不要附上多餘的程式碼：
- 相關檔案的完整路徑清單
- 每個檔案的職責（一句話）
- 最關鍵的程式碼片段（只摘錄有問題或重要的 10-20 行，標明行號）
- 架構摘要（3-5 句話，說明資料流方向）
```

Sub-agent 回傳後，parent agent 只讀取 sub-agent 指出的精確位置，
**不重複搜尋整個 codebase**，保持自身 context 乾淨。

#### 輸出研究文件（格式，直接在對話中顯示）

```
## Research 結果

### 涉及範圍
- `src/main/java/.../Foo.java` — 主要目標，負責 [描述]
- `src/main/java/.../Bar.java` — 被 Foo 依賴，[描述]
- `src/test/java/.../FooTest.java` — 測試覆蓋 [描述]

### 架構摘要
[2-5 句話描述目前的設計，包含資料流方向]

### 發現的問題
- [問題A]（位置：Foo.java:42）
- [問題B]（位置：Bar.java:87-103）
```

**暫停並詢問使用者：** 「Research 完成，以上理解是否正確？有沒有我遺漏的脈絡？」

---

### Phase 2：Plan（計畫）

**目標：** 產出一份詳細計畫，讓使用者可以在不執行程式碼的情況下理解所有變更。

計畫必須包含：

1. **審查結論** — 針對以下四個面向分別說明發現的問題：
   - 邏輯正確性（Logic）
   - 效能（Performance）
   - 安全性（Security）
   - 程式碼風格與可讀性（Style）

2. **變更步驟** — 每個步驟包含：
   - 要修改的檔案與行號
   - **修改前的程式碼片段**（原始碼）
   - **修改後的程式碼片段**（改動後的樣子）
   - 說明「為什麼這樣改」

3. **測試計畫** — 每次變更後如何驗證正確性：
   - 要執行哪些既有測試
   - 是否需要新增測試案例

**Plan 輸出格式：**

```markdown
## Code Review & Refactoring Plan

### 審查結論

**邏輯正確性**
- ⚠️ [問題描述] — `Foo.java:42`
  > 原因：[解釋]

**效能**
- ⚠️ [問題描述] — `Bar.java:87`
  > 原因：[解釋]

**安全性**
- ✅ 未發現明顯安全疑慮 / ⚠️ [問題描述]

**程式碼風格**
- 💡 [建議描述] — `Foo.java:15-20`

---

### 變更步驟

#### 步驟 1：[描述這個步驟的目的]
檔案：`src/main/java/.../Foo.java`，第 42 行

**修改前：**
```java
// 原始程式碼
```

**修改後：**
```java
// 改動後的程式碼
```

原因：[解釋為什麼這樣改更好]

#### 步驟 2：...

---

### 測試計畫
- 執行 `FooTest.java` 中的 `testXxx()` 確認邏輯正確
- 新增測試：[描述需要新增的測試案例]
```

**同時儲存 Plan 為 markdown 檔案：**
儲存到專案根目錄下的 `.ai-plans/YYYY-MM-DD_[簡短描述].md`
（若 `.ai-plans/` 目錄不存在則建立）

**強制暫停** — 計畫完成後說：
> 「計畫已準備好，請 review 以上內容。確認後請說「開始實作」，或告訴我需要調整的地方。」

**不得在使用者確認前進入 Implement 階段。**

---

### Phase 3：Implement（實作）

**目標：** 按照確認後的計畫逐步執行，每步驟後驗證。

執行原則：

- **逐步執行** — 完成一個步驟後，確認無誤再進行下一步
- **不偏離計畫** — 若執行中發現計畫有誤，暫停說明，不自行決定
- **Context 紀律** — 若 codebase 較大，每個步驟只讀取當前需要的檔案，用完即放
- **不過度修改** — 只改計畫中列出的部分，不順手「順便」改其他看起來有問題的地方

實作完成後輸出摘要：

```
## 實作摘要

### 已完成的變更
- `Foo.java:42` — [描述]
- `Bar.java:87-103` — [描述]

### 測試結果
- [測試名稱]：✅ 通過 / ❌ 失敗（附說明）

### 建議後續追蹤
- [若有發現但本次未處理的問題，列在這裡]
```

## Context 管理原則

這些原則來自 Context Engineering 研究，幫助 AI 在複雜 codebase 中保持高品質：

- **不一次讀入所有檔案** — 只讀取當前步驟需要的程式碼，避免 context 過載
- **壓縮而非累積** — 若 codebase 探索過程很長，在進入 Plan 前先整理成精簡的研究文件，而非帶著所有搜尋歷程
- **只標記真正相關的行號** — Plan 中的程式碼片段應精準，不要複製整個類別
- **trajectory 意識** — 若使用者糾正了某個方向，在繼續前先重新確認，不繼續沿著錯誤的路徑走

## Java 特定審查重點

在 Research 和 Plan 階段，特別留意以下 Java 常見問題：

**邏輯：**
- Null 處理（NPE 風險）、Optional 的使用方式
- 例外處理是否恰當（checked vs unchecked）
- 集合操作的邊界條件

**效能：**
- N+1 查詢問題（JPA/Hibernate）
- 不必要的物件建立或 autoboxing
- Stream 操作的效率（避免重複遍歷）
- 同步（synchronized）範圍過大

**安全：**
- SQL injection（JDBC 原生查詢）
- 不安全的反序列化
- 敏感資訊洩漏（日誌中印出密碼、token）
- 輸入驗證缺失

**程式碼風格：**
- 命名是否清晰表達意圖
- 方法是否過長（超過 30 行通常是信號）
- 類別職責是否單一（SRP）
- 重複程式碼（DRY 原則）
